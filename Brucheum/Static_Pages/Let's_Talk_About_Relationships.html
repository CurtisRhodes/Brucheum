Let's Talk About Relationships
Glooma
Relationships make the world go around. Relationships make life worth living. But relationships can be dangerous. Trusting others opens one to getting taken scammed.Be dependent on another puts one in a powerless position. To give your heart risks getting it broken. Relationships lay at heart of computer programming as well. A computer program has to provide a specific solution to some specialized domain be of any value. On a structural level a computer programmer has to make use of tools and existing platforms at all levels. There are interesting principles of computer programming that address the centrality of managing relationships. "Don't talk to strangers", the law of Demeter, is actually a term in computer programming. Loosely coupled coarsely grained is a thing. Dependency injection. Wouldn't that be nice?

They say that in some programming languages you can write a complete program that outputs "Hello World" in one line of code. This may not be entirely true. To output the characters "Hello World" on screen requires a library of graphics code that manipulates pixels to form letter shaped glyphs and paint those characters on the screen, requires more than one line of code. Then you need a library of code to handle keyboard input. Oh, and you need an operating system to process the key presses and the pixel painting. And you probably could use an editor integrated with a compiler and some debugging tools. All told your one line "Hello World" app actually takes many thousands of lines of code, hundreds of thousands, perhaps millions of lines of code. And all but one line of that code is written by others. Your app has to establish relationships with all this external code, and, unfortunately, it needs to be said, your app is dependent on all this code working properly.

We all like to think we are independent. We certainly never want to admit we are dependent on anyone else. As teenagers we rebel against our parents, come to resent them for all they do for us, and struggle to leave the nest and set out on our own. But freedom's just another word for having no one in your life that matters, no one you can count on for help, no one who you have to care for. Maybe freedom isn't all that its cracked up to by libertarians.

Dependencies happen. Dependencies are like the weather. Sometimes you have to put on a coat. But if you are careful and deliberate and aware of the relationships you establish, there are ways to take better control of the situation.

Inside your program, you have much more control over relationships. Let's say you have a written a process called "Clean the Laboratory". Within this process you have created a function called "Put The Cap on the Bottle". Now let's say that you write another module in your app called "Launch A Rocket into Space". Remarkably this module also requires a function that puts a cap on the bottle. What do you do? You could set a reference to your "Clean the Laboratory" process and call the already perfected bottle capping routine. This is the preferred design pattern according to the DRY principle. (Don't Repeat yourself).

Use the well known programming strategy called "cut and paste" and add the bottle capping functionality inside the Launch Rocket code you violate the Department of Redundancy Department prime directive DRY rule, but, it has its advantages. Establishing a relationship between two largely unrelated code bodies just to share one block of code presents its own problems. It may incur expensive performance costs in terms of loading and unloading large chunks of code into memory and then not disposing of them quickly. And there are other problems establishing this relationship might introduce. If both processes happen to have a variable named "Counter" that isn't properly scoped, even if they have nothing to do with each other they could end up stepping all over each other's memory address locations and inadvertently send the rocket ship hurling into the sea. I hate it when that happens. We will talk about scoping and access modifiers in a later article.
