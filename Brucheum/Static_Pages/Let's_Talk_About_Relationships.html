Computer Programming For Girls
Let's Talk About Relationships
by: Cranky Old Man
edit
Dependency Injection, Don't Talk To Strangers. Coarsely Grained Loosely Coupled. All these are actual computer architecture concepts. What do they mean? In this article we will talk about the concepts of relationships in computer programming. Turns out computer programming is all about relationships. This article is intended just to get you thinking about some of the things going on when you write code.
Dependency is bad. We all want and need relationships but they should be managed. When we come to depend on some outside of our control we lose control and have no say over their whims. Ideally a relationship is a good and wonderful thing. But as we all know, relationships can sometimes lead to problems. Unreliable partners can bring us down if they do bad things that is no fault or ours. So we always need to guard against the dangers that come with relationships.
Probably the first program you ever wrote was "Hello World". Programming languages and development environments like to boast they can let you write a program that echoes "Hello Word" to the console screen with one line of code. But is it really just one line of code? Think of all the things behind the scenes that have to take place to get your one line program to work. There has to be a process somewhere that handles the keyboard input to allow you write the words. There has to be a screen graphics process that takes those keystrokes and forms glyphs of letters out of pixels and presents them on your monitor screen. Already we're talking about thousands perhaps millions of lines of code just handle the keyboard and screen graphics aspects of your program. Then you have the compiler that takes your input and boils it down to executable binary code. Then you have the a processor that takes your translated code and executes it through the operating system. By now we really are talking about millions of lines of code. So your one line program isn't really one line after all. Your app has dependencies. It has to form relationships with other chunks of code.
We all like to think we can make all alone on out own. We forget to notice how we depend on the grocery store to deliver our food and all the suppliers who package and deliver our food and the growers and ranchers and pickers and planters and processors who provide the products. We really aren't as self reliant as we like to think. The same applies to the clothes we ware and the shelters we live in. You get the point there are all sorts of dependency relationships out there whether we like it or not. Not all of them we can control.

Inversion of Control

The first thing you have to do is recognize and identify the dependencies you have. We will talk later when we dive deeper into Object Oriented Design about IOC. Basically IOC is just a way of Injecting dependencies into your constructor instead of simply inheriting them or including a using statement. Theoretically this means you could someday replace the dependency with some other object that implements the same interface or create a wrapper around your own version of the object where you can add or modify the functionality it provides. Say you create your own authentication and login controller. You can replace the default object with your custom one.

Practically the say IOC facilitates testing. It allows you to strictly delineate a separation of concerns. Again we'll talk about this later when I can provide links to an article about encapsulation. For now let's just appreciate the concept. Remember, there is nothing wrong with having relationships. You are writing a custom application after all. If you didn't want anything special or unique about your app your users could just use Microsoft Excel Spreadsheet or something and create the functionality themselves. You want to impact your domain in a specialized way so users derive some specific benefit from establishing a relationship with the functionality your app provides.

Loosely Coupled

What sort of relationship do you want? Just friends or something more. To be loosely coupled means that your app doesn't get hopelessly intertwined with some other object beyond the point of graceful separation. If you use string manipulation hundreds of times all over the place you can't just decouple from the String.IO library that easily. Fortunately such base class libraries are usually bullet proof and highly dependable. If there was any problem with one of these core system libraries zillions of people would howl and complain right away. But when get to using some esoteric third party library you might want to be more careful and not get so deeply involved.
The companion concept with Loosely Coupled is "Coarsely Grained". This means you treat the object as a big lump of foreign code that you use explicitly. You don't leave it open forever. You expect it to implement the Idisposable interface and you close it as soon as you are done with it ideally with a using statement. Again if Idisposable and using statements are new to you don't worry for now. These are just more links to future articles I will try to cover.

Don't Talk To Strangers

Once we get a handle on all this talk about relationships we can go one step further. The objects we have established relationships with often have other objects that they have dependency relationships with. It is bad practice to reference these parent objects directly. Just because your aunt's sister in law knows a guy who knows a guy who says he has an in with the governor of some province in Bolivia doesn't mean you should this reference. It is better to create a new object somehow that references this object more directly. That's all for now. I just wanted to give you a brief taste of the concept of relationships in computer app design and get you thinking about when you go about becoming a proficient computer programmer. Thank you for any feedback.
